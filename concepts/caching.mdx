## Overview

quokkajs maintains an **in-memory cache** scoped to the `QuokkaProvider` instance. Every successful query response is stored and served on subsequent identical requests — without hitting the network.

<Info>
  The cache lives inside `QuokkaProvider`. Each provider instance has its own isolated cache. If you have multiple providers, they do not share state.
</Info>

---

## How cache keys are generated

Each cache entry is keyed by a **SHA-256 hash** of the fully-resolved request parameters: URL, method, headers, query params, and body.

Before hashing, the parameters object is **recursively sorted by key**. This ensures two objects with the same properties in different orders produce the same hash — so `{ page: 1, query: 'cats' }` and `{ query: 'cats', page: 1 }` resolve to the same cache entry.

```
Request params → sort keys recursively → JSON.stringify → SHA-256 hash → cache key
```

## Cache hit vs. miss

```
trigger(args)
  │
  ├─ Resolve full request params (baseUrl + url + params + headers)
  │
  ├─ Request already in-flight? ──→ YES ──→ Attach to existing request (no duplicate network call)
  │
  ├─ Generate SHA-256 cache key
  │
  ├─ Cache hit? ──→ YES ──→ Return cached value immediately (no network request)
  │
  └─ Cache miss ──→ Fetch from network
                      │
                      ├─ Success → Store in cache → Return data
                      └─ Failure → Set error state → Throw
```

<Note>
  Cache lookup only happens when `fetchFromCache` is `true` — which is the case for all automatic fetches (`fetchOnRender`, `fetchOnArgsChange`). Manually calling `trigger` always goes to the network.
</Note>

## In-flight request deduplication

If the same query is triggered twice before the first response arrives — for example, `fetchOnRender` fires and the user also clicks a refresh button — only **one** network request goes out. The second call attaches to the first request's in-flight promise and both callers receive the same response when it resolves.

Each caller still gets its own independent loading and error state, so both components show a spinner and both update when the data arrives.

The dedup key is derived from the resolved URL, HTTP method, and request headers. Requests with different `Authorization` headers (different users or auth tokens) are always treated as distinct requests and never deduplicated.

---

## What gets cached

Only **query** responses are cached. Mutations are never cached — they always hit the network.

Each cache entry stores:
- The response data
- The request params (used to re-trigger the query during invalidation)
- The tags the query provided (used for tag matching during invalidation)
- A timestamp and TTL used to compute `isValid` at read time

---

## Cache lifetime (TTL)

Cache entries expire automatically based on a **TTL (time-to-live)**. The default is **30 seconds**.

`isValid` is a computed getter — `Date.now() - timeAdded < ttl` — checked every time the cache is read. Expired entries are treated as misses and trigger a fresh network request.

You can override the TTL per query hook call:

```tsx
useGetTodosQuery('', { fetchOnRender: true, ttl: 60_000 })  // 60 second TTL
useGetTodosQuery('', { fetchOnRender: true, ttl: -1 })       // never expires
```

Pass `-1` to disable expiry entirely — the entry stays valid until a mutation explicitly invalidates it.

Writing to the cache (on a successful fetch) always resets the TTL clock, so a cache entry that is actively being refetched never expires mid-session.

<Info>
  Tag-based invalidation still works independently of TTL. A mutation with a matching `invalidatesTags` will trigger a refetch immediately, regardless of whether the cache entry has expired.
</Info>

## Inspecting the cache

You can access the raw `CacheManager` instance from `useQuokkaContext()` for debugging:

```tsx
import { useQuokkaContext } from 'quokkajs';

function DebugPanel() {
  const { cacheManager } = useQuokkaContext();

  return (
    <pre>
      {JSON.stringify(cacheManager?.apis, null, 2)}
    </pre>
  );
}
```

The `apis` object is keyed by `apiName`, then contains an array of `CacheEntry` objects — each with `id` (the hash), `name` (the hook name), `result`, `tags`, and `isValid`.
